# prd-manager v2.0 — AI 驱动的项目全生命周期管理 Skill

> 从需求到上线，让 AI 成为你的项目管理专家

---

## 🎯 这是什么？

**prd-manager** 是一个专为 AI 协作设计的项目文档管理 Skill，它不仅仅是文档模板库，更是一个**完整的 AI 协作流程引擎**。

通过结构化的文档体系 + 智能对话流程，让 AI 能够：
- 从零搭建完整的项目文档体系
- 引导你把模糊的想法整理成清晰的需求
- 确保代码与文档始终保持一致
- 自动生成测试用例和部署文档
- 记录架构决策和生产事故

**一句话总结**：你只需要描述想法，AI 会一步步引导你完成从需求分析到生产部署的全流程文档。

---

## 💡 为什么需要它？

### 传统方式的痛点

❌ **需求不清晰**：想法很多，但不知道怎么整理成结构化需求  
❌ **文档与代码脱节**：代码改了，文档没更新；文档写了，代码没实现  
❌ **测试被忽略**：开发完才想起来要写测试，结果测试覆盖率很低  
❌ **部署靠记忆**：每次部署都要回忆上次怎么操作的，容易出错  
❌ **事故重复发生**：同样的坑踩了两次，因为没有记录和复盘  

### prd-manager 的解决方案

✅ **AI 主动引导**：通过对话追问，把模糊想法整理成清晰需求  
✅ **文档代码同步**：PRD ↔ 代码双向验证，确保一致性  
✅ **测试驱动开发**：强制要求代码变更必须同步更新测试  
✅ **部署流程标准化**：环境变量清单 + 部署 SOP + 故障排查手册  
✅ **事故记录沉淀**：5 Why 分析 + 预防措施 + 教训总结

## 🚀 核心能力

### 1. 完整的文档生命周期（8 种文档类型）

| 文档类型 | 用途 | 何时生成 |
|---------|------|---------|
| **prd.md** | 产品需求文档（做什么、为什么） | 项目启动 / 新功能 |
| **design.md** | 界面设计文档（长什么样） | 前端项目必须 |
| **dev.md** | 技术开发文档（怎么实现） | 所有项目必须 |
| **plan.md** | 开发计划（何时做、谁来做） | 项目启动 / 大版本 |
| **test.md** | 测试文档（如何验证） | 所有项目必须 |
| **ops.md** | 运维文档（如何部署和运维） | 上线功能必须 |
| **CHANGELOG.md** | 变更日志（版本间变更） | 每次迭代 |
| **CLAUDE.md** | 项目记忆体（AI 可读的项目总纲） | 项目启动 + 持续更新 |

**特点**：
- 每个文档有且只有一个关注点，严禁跨界
- 支持两种目录结构（版本优先 / 类型优先）
- 自动识别项目类型（前端 / 后端 / 全栈）

### 2. AI 协作流程引擎

#### 对话流程设计

```
你：我想做一个任务众包平台

AI：好的，我需要先了解几个问题：
    1. 项目名称是什么？
    2. 技术栈是什么？（前端 / 后端 / 数据库）
    3. 初始版本号？（默认 v0.1.0）

你：项目名 TaskHub，Next.js + NestJS + PostgreSQL

AI：明白了。接下来请描述核心功能模块和优先级...
    [继续引导你整理需求]
    
    [生成 prd.md]
    
    请确认 PRD 是否符合预期，确认后我会继续生成设计和技术文档。

你：确认

AI：[项目审查：读取现有代码]
    [生成 design.md + dev.md]
    
    请确认技术方案是否可行，确认后我会生成开发计划和测试文档。

你：确认

AI：[生成 plan.md + test.md + ops.md + CHANGELOG.md + CLAUDE.md]
    
    ⚠️ 重要提醒：
    1. 开发时必须同步写测试（测试驱动原则）
    2. 部署前必须执行 pnpm test:all 确保全绿
    3. 涉及资金的逻辑必须 100% 确认后才能修改
    
    文档已保存，可以开始开发了。
```

#### 质量门禁

**拒绝场景**（AI 会阻止你）：
- ❌ 跳过测试 → "根据测试驱动原则，任何代码变更必须同步更新测试"
- ❌ 需求模糊 → "需求描述不够清晰，我需要了解：用户角色？核心流程？业务规则？"
- ❌ 不更新文档 → "代码变更前必须先更新 prd.md 或 dev.md"
- ❌ 资金逻辑未确认 → "涉及资金的逻辑必须 100% 确认后才能修改"

**警告场景**（AI 会提醒但允许）：

- ⚠️ 后端项目跳过 design.md → "后端项目通常不需要，但如果有管理后台建议补充"
- ⚠️ 开发阶段跳过 ops.md → "可以暂时跳过，但上线前必须补充"

#### 自检清单

AI 生成文档后会自动检查：
- prd.md：用户角色、验收标准、优先级、无技术细节
- dev.md：数据库变更、API 影响、向后兼容、基于实际代码
- test.md：测试策略、测试用例、检查清单、覆盖率目标
- ops.md：环境变量、部署流程、故障排查、回滚方案

### 3. 测试驱动开发体系

#### 测试金字塔

```
           /\
          /  \  E2E 测试（10%）
         /____\
        /      \  集成测试（30%）
       /________\
      /          \  单元测试（60%）
     /____________\
```

#### 强制规则

| 代码变更类型 | 必须同步的测试 |
|-------------|---------------|
| 新增功能 | 新增单元测试 + 集成测试 + 更新 test.md |
| Bug 修复 | 新增回归测试 + 更新 test.md |
| 重构 | 确保现有测试通过 + 更新测试文档 |
| API 变更 | 更新接口测试 + 更新 test.md |

#### 静态扫描脚本

自动生成审计脚本（放在 `scripts/` 目录）：
- `audit-feature-completeness.mjs`：PRD 覆盖率、静默吞错、service stub
- `audit-db-indexes.mjs`：金额精度、高频字段索引
- `audit-security-perf.mjs`：硬编码密钥、SQL 注入、N+1 查询
- `comprehensive-test.mjs`：字段约束、计算准确性、文档一致性

#### 部署前检查清单

```markdown
- [ ] pnpm test:all 全绿
- [ ] 环境变量已配置（对照 ops.md）
- [ ] 数据库迁移脚本已验证
- [ ] 回滚方案已准备
- [ ] 本地起服务 → seed 真实数据 → 走一遍 happy path + edge cases
```

### 4. 运维部署体系

#### 环境变量管理

```markdown
| 变量名 | 必填 | 默认值 | 说明 | 获取方式 |
|--------|------|--------|------|---------|
| DATABASE_URL | ✅ | - | PostgreSQL 连接 | Supabase 控制台 |
| TURNSTILE_SECRET_KEY | ✅ | - | Cloudflare Turnstile | dash.cloudflare.com |
| NODE_ENV | ✅ | development | 运行环境 | 手动设置 |
```

**原则**：所有环境变量必须在 ops.md 中文档化

#### 部署流程 SOP

```markdown
### 前置条件
- [ ] 所有测试通过
- [ ] 代码已合并到 main 分支
- [ ] 环境变量已配置
- [ ] 数据库迁移脚本已准备

### 部署步骤
1. 构建：bash scripts/deploy.sh
2. 上传：自动上传到服务器
3. 迁移：npx prisma migrate deploy
4. 重启：pm2 restart app
5. 验证：访问 /health 确认 200

### 回滚步骤
1. 代码回滚：git revert <commit> && git push
2. 重新部署：bash scripts/deploy.sh
3. 数据库回滚：执行回滚 SQL 脚本
4. 验证：确认服务恢复正常
```

#### 故障排查手册

```markdown
### 问题：用户无法登录，提示"登录失败"

排查步骤：
1. 检查后端日志：sudo pm2 logs api --err
2. 查找 POST /api/auth/login 请求
3. 常见原因：
   - 数据库连接失败
   - JWT_SECRET 未配置
   - 用户不存在或密码错误

解决方案：
- 检查 DATABASE_URL 是否正确
- 检查 JWT_SECRET 是否配置
- 检查用户表是否有数据
```

### 5. 事故记录与复盘

#### 事故记录结构

```markdown
# 事故记录：用户无法领取任务

**发现时间**：2026-05-02 23:00
**影响范围**：生产环境，100 个用户
**严重程度**：P1（核心功能异常）
**状态**：✅ 已修复

## 问题描述
用户点击"领取任务"按钮后，提示"领取失败，请重试"

## 根因分析（5 Why）
1. 为什么失败？后端返回 400 错误
2. 为什么返回 400？Turnstile token 验证失败
3. 为什么验证失败？前端未传递 token
4. 为什么未传递？前端代码改了，后端没部署
5. 根本原因：部署流程不完整，前后端未同步部署

## 修复方案
1. 部署后端代码
2. 验证 Turnstile 验证流程
3. 新增回归测试

## 预防措施
短期（1 周内）：
- [ ] 新增回归测试
- [ ] 更新部署文档

中期（1 个月内）：
- [ ] 引入 CI/CD 自动部署
- [ ] 前后端部署原子化

## 教训
1. 前后端必须同步部署
2. 部署前必须跑完整测试
3. 关键功能必须有监控告警
```

### 6. PRD ↔ 代码双向验证

#### 正向验证：PRD → 代码

确保 PRD 中的每个需求都有对应的代码实现

```javascript
const prd = readPRD('docs/prd/v1.0.0/prd.md');
const dev = readDev('docs/dev/v1.0.0/dev.md');

for (const requirement of prd.requirements) {
  const hasCode = dev.fileChanges.some(f => f.relates(requirement.id));
  const hasTest = test.testCases.some(t => t.covers(requirement.id));
  if (!hasCode || !hasTest) {
    fail(`需求 ${requirement.id} 未完整实现`);
  }
}
```

#### 反向验证：代码 → PRD

确保代码中的每个功能都有 PRD 依据

```javascript
const controllers = scanControllers('apps/api/src/modules/');
for (const controller of controllers) {
  const inPRD = prd.requirements.some(r => r.covers(controller.feature));
  if (!inPRD) {
    warn(`API ${controller.endpoint} 缺少 PRD 依据`);
  }
}
```

### 7. CLAUDE.md 项目记忆体

#### 架构决策记录（ADR）

```markdown
### ADR-001：选择 PostgreSQL 而非 MongoDB

**背景**：需要存储用户、任务、订单等数据

**决策**：使用 PostgreSQL

**理由**：
- 需要事务支持（资金相关）
- 需要复杂查询（多表 JOIN）
- 需要数据一致性保障

**影响**：
- 使用 Prisma ORM
- 数据库迁移用 prisma migrate
- 需要学习 SQL
```

#### 给 AI 的元规则

```markdown
### 行为准则

- 默认假设需要测试：看到"修一下 X"，默认假设需要同步加测试
- 不要省略测试：用户没说"不用测试"之前，不要省
- 改完后必须跑测试：改完后必须执行 pnpm test:all 确认没引入回归
- 端到端验证：本地起服务 → seed 真实数据 → 走一遍 happy path
- 不要静默吞错：catch 块必须 console.error 或 showToast
```

#### 版本号规则

| 变更类型 | 版本号变化 | 示例 | 触发条件 |

| **Minor** | v1.7.x → v1.8.0 | 新增功能、新增模块 | 向后兼容 |
| **Patch** | v1.7.7 → v1.7.8 | bug 修复、性能优化 | 向后兼容 |

### ✅ 适合

- **个人开发者**：想要规范化自己的项目开发流程
- **小团队**：需要统一文档规范，提升协作效率
- **AI 重度用户**：希望 AI 能更好地理解项目上下文
- **注重质量**：希望通过测试驱动、文档驱动提升代码质量
- **长期维护项目**：需要记录架构决策、事故教训

### ❌ 不适合

- **一次性脚本**：不需要文档的临时项目
- **纯前端静态页面**：不需要后端、数据库的简单页面
- **不想写文档**：觉得写文档是浪费时间的人

## 📥 如何安装？

### 方式 1：通过 Claude Code Skill 目录安装

```bash
git clone https://github.com/wlzh/prd-manager ~/.claude/skills/prd-manager
```

### 方式 2：手动安装

1. 下载 Skill 文件
2. 放到 `~/.claude/skills/prd-manager/` 目录
3. 重启 Claude Code

### 验证安装

```bash
# 在 Claude Code 中输入
/prd-manager 帮我创建一个 v0.1.0 的项目文档

# 如果 AI 开始询问项目信息，说明安装成功
```

## 🎓 快速上手

### 第一步：创建项目文档

```
你：/prd-manager 我想做一个博客系统，Next.js + PostgreSQL

AI：好的，我需要先了解几个问题：
    1. 项目名称是什么？
    2. 初始版本号？（默认 v0.1.0）
    3. 核心功能模块有哪些？

你：项目名 MyBlog，版本 v0.1.0，核心功能是文章发布、评论、用户管理

AI：[生成完整文档体系]
    [提醒测试驱动原则]
```

### 第二步：开始开发

按照 `plan.md` 的任务清单开始开发，记住：
- ✅ 每个功能都要写测试
- ✅ 部署前跑 `pnpm test:all`
- ✅ 遇到问题记录到 `incidents/`

### 第三步：功能迭代

```
你：/prd-manager 在 v0.1.0 基础上新增搜索功能

AI：[识别变更类型：Minor]
    [建议版本号：v0.2.0]
    [生成增量文档]
```
