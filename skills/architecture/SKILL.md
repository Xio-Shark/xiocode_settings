# Architecture

## Overview

Use this skill to turn requirements and constraints into a concrete architectural decision with explicit tradeoffs.

## Rules

- Start from requirements and constraints, not favorite patterns.
- Prefer the simplest architecture that can satisfy the known needs.
- Consider tradeoffs explicitly instead of pretending a design is free.
- Capture important decisions in a concise ADR-style rationale.

## When to use

- system design decisions
- pattern selection
- boundary and interface design
- architecture review

## Workflow

1. Clarify requirements, constraints, and team context.
2. Identify the main architectural decision points.
3. Compare realistic options and choose one.
4. Explain why it was chosen, what it costs, and what it enables.
5. Produce a concrete implementation-oriented architecture summary.

## Output expectations

- requirements and constraints
- chosen architecture
- key tradeoffs
- major boundaries/components
- ADR-style rationale
